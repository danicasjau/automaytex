AUTOMAYTEX: The Comprehensive Guide
Automated Bake–Diffuse–Reproject Texturing Pipeline for Autodesk Maya 2026
Author: Daniel Casadevall Jauhiainen Platform: Autodesk Maya 2026 / 2025 Version: 1.0-alpha Renderer: Arnold API (Python) Format: Maya command API Language: Python 3.10+
Abstract
Automaytex is a state-of-the-art Maya plugin designed to automate AI-driven surface texture generation seamlessly within the authoring environment. The system intelligently captures geometric data (surface normals and depth) from four optimally positioned cameras using a tetrahedral projection strategy. This data is composited into a 2×2 atlas, which conditions a Diffusion Model (SDXL, Flux, or SD1.5) via ControlNet to produce coherent, high-fidelity textures. Finally, the generated texture is seamlessly reprojected onto the original UV layout using a custom depth-weighted blending algorithm. The pipeline automatically manages hardware resources, adjusting quantization levels dynamically to ensure stability across diverse GPU configurations.

1. Overview

Automaytex is engineered for look-development artists and 3D generalists who need to rapidly prototype and apply complex, high-quality surface materials to 3D assets without ever leaving Autodesk Maya. By abstracting away the complexities of AI inference, the plugin wraps a complete Stable Diffusion/ControlNet pipeline—geometry extraction, image compositing, AI generation, and precise UV reprojection—behind an intuitive GUI panel.

Core Philosophy: The workflow assumes your mesh already has a valid, non-overlapping UV layout. Automaytex does not alter your UVs or modify your geometry; it operates non-destructively by executing analytical renders to extract shape context, feeding that context to an AI, and then mapping the AI's output back to your existing UV space.

Key Capabilities

  • Non-Destructive Workflow: A snapshot of the original UVs is taken before execution. The geometry remains untouched; only a new aiStandardSurface material is applied upon completion.
  • Resolution Independence: Internal atlas resolution (for AI generation) and final texture output resolution are independently configurable.
  • Hardware-Adaptive Execution: The system automatically probes system VRAM on startup and scales model precision (FP16, INT8, INT4, or CPU offload) to prevent memory crashes.
  • Model Agnosticism: Supports Flux, SDXL, and Stable Diffusion 1.5 through a unified inference interface, allowing artists to leverage the best models available.

3. Comprehensive Installation Guide

Installing Automaytex requires configuring Maya's internal Python environment to support complex deep-learning libraries. Please follow these steps precisely.

Step 1: Clone the Repository

Download the Automaytex source code to your desired project directory.

# Open your terminal or command prompt
git clone https://github.com/youruser/automaytex.git
cd automaytex

Step 2: Install Python Dependencies into Maya

You must use Maya's bundled Python interpreter (mayapy.exe on Windows or mayapy on macOS/Linux). Do not use your system's global Python environment, as Maya will not be able to read those modules. 

# Locate mayapy (usually C:\Program Files\Autodesk\Maya2026\bin\mayapy.exe)
# Install PyTorch with CUDA 12.1 support:
mayapy -m pip install torch torchvision --index-url https://download.pytorch.org/whl/cu121

# Install diffusers and quantization libraries:
mayapy -m pip install diffusers transformers accelerate bitsandbytes pillow

Step 3: Download and Configure Model Weights

Due to file size limits, model weights are not bundled with the source code. You must download the necessary checkpoint files (e.g., from HuggingFace or Civitai) and place them in the weights/ directory.

  • Download SDXL Base 1.0 or your preferred model.
  • Download the corresponding ControlNet Normal Map model.
  • Edit config.yaml to point to the absolute or relative paths of these downloaded weights.

Step 4: Load the Plugin in Maya

  1. Launch Autodesk Maya 2026.
  2. Navigate to Window → Settings/Preferences → Plug-in Manager.
  3. Click the Browse button and select src/main.py from the cloned repository.
  4. Check the Loaded box. (Optionally check Auto load to have it launch with Maya).
  5. The Automaytex tool shelf button will now appear on your UI, ready to be used.

4. Pipeline Architecture

The Automaytex pipeline is meticulously structured into three sequential, decoupled phases. This modularity ensures that if one phase fails, the intermediate data (stored in a temporary directory) can be inspected, and the process can be debugged independently.

Pre-computation Step: UV Snapshotting. Before Phase 1 begins, the system serializes the target mesh's UV coordinates to a file named uv_manifest.json. This guarantees that Phase 3 maps the generated pixels back exactly as the geometry existed at launch time, rendering the process resilient to accidental mesh edits while the generation runs in the background.

5. Tetrahedral Viewpoint Strategy

How does the AI know the shape of the 3D object? Early versions of Automaytex utilized a standard 6-plane orthographic projection (a box mapping from +X, -X, +Y, -Y, +Z, -Z). This approach was highly inefficient: opposing cameras captured near-duplicate visual information, and arranging 6 images into a single square atlas created rectangular sub-images that broke the aspect-ratio conditioning of the Diffusion Model.

Automaytex resolves this via the Tetrahedral Strategy.

Mathematical Implementation

By defining a regular tetrahedron perfectly inscribed within the bounding sphere of the mesh, we determine four camera vectors that are maximally and uniformly separated. The angle between any two camera vectors is derived as:

(1) $$\theta = \arccos\!\left(-\tfrac{1}{3}\right) \approx 109.47°$$

This layout ensures approximately ~12% average visual overlap between cameras, compared to ~35% with the outdated box projection method.

The 2x2 Atlas

The four camera views are stitched seamlessly into a single 2×2 square image atlas. This single square atlas is perfectly suited for Diffusion Models, which are inherently trained on square aspect ratios (e.g., 512x512, 1024x1024). The same quadrant layout is applied to the depth data to ensure mathematical parity during blending.

6. Phase 1 — Geometry Bake (Arnold)

In Phase 1, Automaytex algorithmically instantiates four Arnold camera nodes based on the tetrahedral vectors. It then interfaces directly with the Arnold Core API to execute highly optimized, flat-shaded renders.

For each camera, it extracts two vital pieces of information (AOVs):

  • Normals AOV (N): Captures the world-space surface normal direction of every pixel, encoded into RGB color space. This provides the ControlNet with the structural definition of the object.
  • Depth AOV (Z): Captures the distance from the camera to the surface pixel. It is normalized and tone-mapped. This map is crucial for Phase 3, dictating how different views blend together cleanly.

The output of Phase 1 is 8 individual EXR files (4 normal maps, 4 depth maps), which are then composited into two master 2×2 PNG atlases.

7. Phase 2 — AI Diffusion Inference

This is where the generative AI takes over. The system loads the designated Stable Diffusion pipeline alongside the ControlNet model. The Normals Atlas generated in Phase 1 is passed as the structural anchor.

The model processes the user's text prompt (e.g., "weathered bronze with green patina") alongside generation presets loaded from rules.json. By applying Classifier-Free Guidance (CFG), the model iterates over the noise to produce a fully textured 2x2 image atlas.

Understanding ControlNet Scale: If the controlnet_scale (default 0.85) is too high, the texture may look overly rigid or fail to adopt organic patterns. If it's too low, the AI might hallucinate details that break the physical geometry.

8. Phase 3 — UV Reprojection & Blending

Phase 3 resolves the fundamental problem of 3D texturing: how to wrap a 2D image back onto a complex 3D shape without visible seams.

Automaytex reads the 2x2 textured output atlas. Using the uv_manifest.json from the pre-computation step, it maps the pixels from the atlas back to the UV space of the mesh. Because the four cameras overlap at the edges, Automaytex applies a sophisticated depth-weighted blending algorithm.

(2) $$\mathbf{C}(u,v) = \frac{\displaystyle\sum_{i} w_i(u,v)\cdot \mathbf{C}_i(u,v)}{\displaystyle\sum_{i} w_i(u,v)}$$

Here, $w_i$ represents the weight based on the camera's depth map. The algorithm favors pixels that were closer to the camera (higher clarity) and smoothly blends the borders between overlapping views, effectively erasing harsh transition seams.

Finally, the synthesized texture is written to disk as a high-resolution EXR file, and a brand new aiStandardSurface material node is created, wired, and assigned to the mesh in Maya.

9. Dynamic VRAM Quantization

Loading multi-gigabyte AI models into GPU memory alongside Maya and Arnold is highly prone to crashing. Automaytex includes an intelligent memory management system that queries torch.cuda.mem_get_info() on initialization.

Depending on the available VRAM, the engine dynamically applies Quantization via the bitsandbytes library to shrink the memory footprint of the weights:

Detected VRAM Quantization Level Impact on Output
≥ 16 GB FP16 (Full Precision) None. Maximum quality.
8 – 15 GB INT8 (8-bit Precision) Extremely minor artifacting; virtually imperceptible.
4 – 7 GB INT4 (4-bit Precision) Noticeable softening in micro-details, but prevents crashing.
< 4 GB / CPU CPU Offload Very slow generation times. Only used as an absolute fallback.

10. How the Code Works: Maya Integration (main.py)

The main.py script serves as the bridge between Maya's environment and the AI engine. It is strictly responsible for UI, Maya scene graph manipulation, and threading.

  • initializePlugin() / uninitializePlugin(): Standard Maya API hooks to register the plugin and attach the shelf buttons.
  • UI Generation: Built using PySide2, creating the dockable or floating Qt panel that the artist interacts with.
  • Worker Threads: To prevent Maya from freezing completely during the heavy AI diffusion process (which takes 30-120 seconds), main.py spawns a QThread. The AI process runs asynchronously and reports progress back to the main thread via Qt Signals.
  • apply_texture(): The final callback that utilizes cmds.shadingNode to construct the Arnold shader network and assign it to the selected object.

11. How the Code Works: Inference Engine (engine.py)

engine.py encapsulates the entirety of the pipeline logic. It is completely decoupled from Maya's UI framework, meaning it can theoretically be executed via a standalone headless Python script for batch processing.

  • run_pipeline(mesh, prompt, config): The primary orchestrator function that calls Phase 1, 2, and 3 sequentially.
  • build_camera_matrix(): Contains the linear algebra required to position the cameras at the exact tetrahedral vertices.
  • run_diffusion(): Initializes the HuggingFace Diffusers pipeline, applies the necessary bitsandbytes quantization flags, passes the latents, and executes the denoising loop.
  • reproject_to_uv(): Iterates through the output image, referencing the UV manifest to calculate the weighted pixel blending logic mathematically.

12. Configuration & Presets (config.yaml / rules.json)

Automaytex relies on two distinct configuration files to allow artist flexibility without requiring Python edits. These files define everything from hardware limits to material properties.

config.yaml

Controls system-level paths, absolute resolutions, and hardware override flags.

# Example config.yaml snippet
models:
  sdxl: "D:/models/sdxl-base-1.0"
  controlnet: "D:/models/controlnet-normals-sdxl"
render:
  atlas_resolution: 1024
  output_resolution: 2048
Configuration Parameters
Parameter Description
models.sdxl / flux / sd15 The absolute or relative path to the diffusion model checkpoint directories on your disk.
models.controlnet The path to the ControlNet normal map conditioning model weights.
quantization.force_fp16 If true, ignores the VRAM check and forces full 16-bit precision (Warning: may crash if VRAM is insufficient).
quantization.vram_thresholds Defines the memory thresholds (in GB) at which the system drops from FP16 → INT8 → INT4.
render.atlas_resolution The resolution of the internal 2x2 composited image sent to the AI (e.g., 1024x1024). Lowering this saves VRAM.
render.output_resolution The resolution of the final reprojected texture applied to your mesh in Maya (e.g., 2048 or 4096).
render.aov_samples Anti-aliasing quality for the Arnold geometry bake. Higher values (e.g., 4 or 8) produce cleaner depth maps for blending.
paths.temp_dir Directory where intermediate EXR and JSON files are temporarily stored.
paths.output_dir Directory where the final generated AI textures are permanently saved.

rules.json

Defines the drop-down presets available in the UI. Each preset encapsulates prompt engineering and CFG tuning for specific material types.

// Example rules.json snippet
{
  "rules": [
    {
      "id": "corroded_copper",
      "label": "Corroded Copper Patina",
      "base_prompt": "aged oxidized copper, heavy green patina, metallic underlay...",
      "negative_prompt": "smooth, shiny, perfect",
      "controlnet_scale": 0.82,
      "cfg_scale": 8.0,
      "steps": 30
    }
  ]
}
Preset Parameters
Parameter Description
id A unique internal identifier string for the preset (no spaces).
label The human-readable name that appears in the Maya UI dropdown menu.
base_prompt The positive text description of the material. Guides the AI on what to generate.
negative_prompt Things you explicitly want the AI to avoid (e.g., "blur, seams, unrealistic").
controlnet_scale How strictly the AI follows the normal map (0.0 to 1.0). ~0.85 is recommended.
cfg_scale How strictly the AI follows the text prompt. Higher values (7.0 - 12.0) increase adherence but may cause color burn.
steps The number of denoising steps the AI performs. ~30 is standard. More steps take longer but yield higher quality.

These files are highly useful for debugging rendering failures.

14. Troubleshooting Guide

Maya Crashes Upon Pressing "Generate"

Cause: Your GPU completely ran out of VRAM, causing the PyTorch environment to crash Maya.
Solution: Open config.yaml and ensure quantization.force_fp16 is set to false. You can also manually adjust the VRAM thresholds to force INT4 quantization, or lower the render.atlas_resolution to 512.

Error: ModuleNotFoundError: No module named 'torch'

Cause: Dependencies were installed to your system's global Python, not Maya's internal environment.
Solution: Run the pip install commands strictly using mayapy.exe as outlined in the Installation section.

The Generated Texture has Horrible Seams and Glitches

Cause: Your object's UV layout is poor (e.g., overlapping islands, stretched faces, or islands rotated extremely far from the camera axis).
Solution: Ensure your UVs are fully laid out and non-overlapping. Alternatively, increase render.aov_samples in config.yaml to generate a cleaner depth map, which improves blending accuracy.

The Texture Appears Very Generic and Lacks Detail

Cause: The ControlNet constraint is overpowering the text prompt.
Solution: Open rules.json and reduce the controlnet_scale slightly (e.g., from 0.85 to 0.75), and potentially increase the cfg_scale to encourage the AI to adhere more strongly to the text prompt.

15. Glossary of Terms

Term Definition
AOV (Arbitrary Output Variable) A specific render pass isolated by Arnold. Automaytex uses Normals and Z-Depth AOVs to analyze the geometry shape.
Atlas A composite image holding multiple sub-images. Automaytex tiles 4 camera views into a single 2x2 grid to feed to the AI model.
ControlNet A neural network extension that allows spatial control (like geometry outlines or normals) over a Stable Diffusion generation.
CFG (Classifier-Free Guidance) A parameter defining how strictly the AI should adhere to the text prompt versus creative freedom. Higher values = stricter adherence.
Quantization The process of mathematically rounding down the weights of a neural network (from FP16 to INT8 or INT4) to save VRAM, with minimal quality loss.
Tetrahedral Projection A method of mapping a 3D object from 4 distinct, equally spaced angles based on the vertices of a regular tetrahedron, minimizing camera overlap.
